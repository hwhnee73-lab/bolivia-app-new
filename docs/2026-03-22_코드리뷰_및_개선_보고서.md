## 1. 프로젝트 개요

| 항목 | 내용 |
|---|---|
| **백엔드** | Spring Boot 3.2.4 / Java 17 / Gradle 8.5 |
| **프론트엔드** | React 18 (CRA) / Axios / React Router v6 / i18next |
| **DB** | MySQL 8.0 (15개 테이블) |
| **인프라** | Docker Compose (db + backend + frontend) / Nginx 리버스 프록시 |
| **도메인** | `redidencial.cor-web.com` (TLS 미적용 상태) |

### 백엔드 패키지 구조

```
com.bolivia.app/
├── config/        (3) SecurityConfig, WebConfig, JpaAuditingConfig
├── controller/   (11) Auth, Bill(2), Finance, Maintenance, Payment, Reservation, ...
├── dto/          (13+) DTO 클래스
├── entity/        (7) User, Household, Bill, Payment, Announcement, RefreshToken, BaseEntity
├── exception/     (5) GlobalExceptionHandler, ErrorResponse, ResourceNotFoundException,
│                      InvalidTokenException, DuplicateResourceException ← 신규
├── repository/    (5) User, Bill, Household, Announcement, RefreshToken
├── security/      (6) JWT 관련 + CurrentUserId resolver
└── service/      (14) 비즈니스 로직 서비스
```

### 4.3 GlobalExceptionHandler 리팩토링

**파일**: `backend/src/main/java/com/bolivia/app/exception/GlobalExceptionHandler.java`

**추가된 핸들러:**
```java
@ExceptionHandler(InvalidTokenException.class)    // → 401 Unauthorized
@ExceptionHandler(DuplicateResourceException.class) // → 409 Conflict
```

**제거된 코드** — `RuntimeException` 핸들러의 문자열 매칭 로직:
```diff
-if (message.contains("User not found") || message.contains("not found")) {
-    status = HttpStatus.NOT_FOUND;
-} else if (message.contains("Access denied") || message.contains("token")) {
-    status = HttpStatus.UNAUTHORIZED;
-} else if (message.contains("Invalid") || message.contains("expired")) {
-    status = HttpStatus.BAD_REQUEST;
-}
+// RuntimeException은 순수 500 INTERNAL_SERVER_ERROR로 통일
```

### 4.4 AuthService / AuthController 예외 교체

**AuthService.java** — 6곳 수정:

| 기존 | 변경 |
|---|---|
| `throw new RuntimeException("Invalid refresh token")` | `throw new InvalidTokenException("Invalid refresh token")` |
| `throw new RuntimeException("Refresh token expired")` | `throw new InvalidTokenException("Refresh token expired")` |
| `throw new RuntimeException("Refresh token not found")` | `throw new InvalidTokenException("Refresh token not found")` |
| `throw new RuntimeException("User not found")` (2곳) | `throw new ResourceNotFoundException("User", ...)` |

**AuthController.java** — 1곳 수정:

| 기존 | 변경 |
|---|---|
| `throw new RuntimeException("Refresh token not found")` | `throw new InvalidTokenException("Refresh token not found in cookie")` |

### 4.5 🐛 JwtTokenProvider 보안 버그 수정

**파일**: `backend/src/main/java/com/bolivia/app/security/JwtTokenProvider.java`

**문제**: `validateToken()` 메서드가 `java.lang.SecurityException`을 catch하고 있었으나, JJWT 라이브러리는 `io.jsonwebtoken.security.SecurityException`(→ `JwtException` 상속)을 throw합니다. 두 클래스는 전혀 다른 상속 체계이므로, 변조/다른키 토큰에서 예외가 전파되어 **프로덕션에서 500 Internal Server Error**로 이어질 수 있는 버그였습니다.

```diff
-} catch (SecurityException ex) {
+} catch (io.jsonwebtoken.security.SecurityException ex) {
     log.error("Invalid JWT signature");
```

### 4.7 기존 테스트 수정

#### ResidentApiTests.java — Mock SecurityContext 추가

**문제**: `@AutoConfigureMockMvc(addFilters = false)`로 보안 필터를 비활성화하면 `SecurityContextHolder`에 `Authentication`이 null. `@CurrentUserId` resolver가 `BadCredentialsException` 발생.

**해결**: `@BeforeEach`에서 Mock SecurityContext 설정:
```java
@BeforeEach
void setUpSecurityContext() {
    // 1. 테스트 User 엔티티 생성 (Reflection으로 ID=1L 설정)
    // 2. SecurityContextHolder에 UsernamePasswordAuthenticationToken 설정
    // 3. UserRepository.findByUsernameOrEmail mock 설정
}
```

#### App.test.jsx — authService mock 추가

**문제**: `AuthProvider`가 마운트 시 `authService.refreshToken()` 호출 → JSDOM에서 실제 HTTP 요청 실패.

**해결**: authService 전체를 mock하여 즉시 실패 반환:
```javascript
jest.mock("../services/authService", () => ({
  __esModule: true,
  default: {
    refreshToken: jest.fn().mockRejectedValue(new Error("no session")),
    // ...
  },
}));
```

#### http.test.js — jest.isolateModules 패턴 적용

**문제**: `http.test.js`의 axios mock이 `App.test.jsx`의 전역 axios mock과 충돌하여 함께 실행 시 실패.

**해결**: `jest.isolateModules`로 각 테스트에서 독립된 모듈 환경으로 http.js 로드.
